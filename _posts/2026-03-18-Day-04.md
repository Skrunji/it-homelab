# Day 4

Today I built my first working dashboard using Streamlit, connected to the Sakila database. This was the first time everything started to feel like an actual application instead of just running isolated SQL queries.

I started by installing Streamlit and setting up a basic dashboard.py file. After wiring up a database connection using environment variables, I launched the app and confirmed it was working by seeing a blank page with a title.

From there, I built out three reports that would be useful in a real video rental business:

* Most popular genres

* Overdue rentals

* Top customers by rental count

The first report (most popular genres) was straightforward. I joined the rental, inventory, film, film_category, and category tables, grouped by genre, and counted total rentals. This produced a clean, working table in the dashboard and confirmed the full pipeline (DB → Python → Streamlit) was functioning.

For the overdue rentals report, I initially misunderstood the meaning of `return_date`. I treated it like a due date, but it's actually the actual return timestamp. That led to incorrect logic at first `(return_date < NOW())`, which only showed already-returned rentals.

I corrected this by recognizing that:

`return_date IS NULL` means the item was never returned

NULL in SQL represents absence of a value, not zero or empty

From there, I improved the logic further by calculating a true due date using:
```
DATE_ADD(r.rental_date, INTERVAL f.rental_duration DAY)
```
This allowed me to filter for rentals that were both:

* Not returned `(return_date IS NULL)`

* Past their expected due date

I validated the logic by temporarily replacing `NOW()` with a historical date (`'2005-09-01'`) since the dataset is old. This confirmed the query worked correctly before switching back to live logic.

I also improved the output by:

* Removing unnecessary columns (like `return_date` since it's always NULL here)

* Adding a calculated `Due_Date` field for clarity and usability

The third report (top customers) worked on the first try, but I ran into a Python indentation issue caused by VSCode autocomplete. The code looked fine but didn’t execute correctly due to extra indentation. I fixed it manually line-by-line, which reinforced how strict Python is about whitespace.

Later, I encountered a warning from pandas about using mysql-connector-python. Instead of ignoring it, I switched to SQLAlchemy for a more standard and scalable approach. This required updating the connection logic to use a connection string, but the rest of the code stayed the same. After the change, the warning disappeared and everything still worked.

I also learned that SQLAlchemy manages connections differently, so calling conn.close() was no longer necessary.

By the end of the session, I had a fully working dashboard with:

* Interactive buttons

* Real database queries

* Clean, usable outputs

* Improved query logic based on actual business needs

The biggest takeaway today was understanding the difference between raw data and meaningful data. A query that runs isn’t enough. It has to answer a real question in a way that someone can act on.

This is the first time this project has felt like something that could exist in a real environment instead of just a learning exercise.
